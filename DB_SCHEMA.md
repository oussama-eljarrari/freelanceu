# Database Architecture & SQLite Schema Findings

## Brainstorming Session Findings
1. **Normalization over Snapshots**: Initially, the backend stored data loosely like a NoSQL database (e.g., storing a snapshot of the seller directly inside the gig). To move to a relational SQL database, we must normalize the data. 
2. **Category vs Tags vs GigTags**:
   - **Category**: A strict, predefined classification (1-to-Many).
   - **Tags**: Dynamic, granular keywords representing specific skills or niches.
   - **GigTags (Join Table)**: Required in relational databases to resolve the Many-to-Many relationship between Gigs and Tags without using arrays.
3. **Decoupling Data Storage from UI**: The database structure will be strictly relational to guarantee data integrity, but the APIs will still return the rich, nested data payloads the frontend expects.

---

## SQLite Database Schema Draft

Since SQLite does not support native `ENUM` or native `UUID` gen_random capabilities by default, UUIDs are managed via `TEXT` types (generated by the backend ORM) and Enums are enforced using `CHECK` constraints.

```sql
-- Users (Clients, Freelancers, Admins)
CREATE TABLE users (
    id TEXT PRIMARY KEY, -- UUID generated by application
    name TEXT NOT NULL,
    email TEXT UNIQUE NOT NULL,
    password_hash TEXT NOT NULL,
    avatar TEXT,
    role TEXT DEFAULT 'client' CHECK(role IN ('client', 'freelancer', 'admin')),
    bio TEXT,
    joined_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    rating REAL DEFAULT 0.00,
    total_reviews INTEGER DEFAULT 0
);

-- Gigs
CREATE TABLE gigs (
    id TEXT PRIMARY KEY,
    seller_id TEXT NOT NULL,
    title TEXT NOT NULL,
    description TEXT NOT NULL,
    category TEXT NOT NULL CHECK(category IN ('Graphic Design', 'Translation', 'Development', 'Writing', 'Tutoring', 'Video Editing', 'Photo Editing', 'Audio & Voice')),
    price REAL NOT NULL,
    delivery_days INTEGER NOT NULL,
    thumbnail TEXT,
    rating REAL DEFAULT 0.00,
    total_reviews INTEGER DEFAULT 0,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY(seller_id) REFERENCES users(id) ON DELETE CASCADE
);

-- Tags
CREATE TABLE tags (
    id TEXT PRIMARY KEY,
    name TEXT UNIQUE NOT NULL
);

-- Gig Tags (Many-to-Many Bridge)
CREATE TABLE gig_tags (
    gig_id TEXT NOT NULL,
    tag_id TEXT NOT NULL,
    PRIMARY KEY (gig_id, tag_id),
    FOREIGN KEY(gig_id) REFERENCES gigs(id) ON DELETE CASCADE,
    FOREIGN KEY(tag_id) REFERENCES tags(id) ON DELETE CASCADE
);

-- Orders
CREATE TABLE orders (
    id TEXT PRIMARY KEY,
    gig_id TEXT NOT NULL,
    client_id TEXT NOT NULL,
    freelancer_id TEXT NOT NULL,
    status TEXT DEFAULT 'pending' CHECK(status IN ('pending', 'in_progress', 'delivered', 'completed', 'cancelled')),
    price REAL NOT NULL,
    requirements TEXT,
    delivery_deadline DATETIME NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY(gig_id) REFERENCES gigs(id) ON DELETE SET NULL,
    FOREIGN KEY(client_id) REFERENCES users(id) ON DELETE CASCADE,
    FOREIGN KEY(freelancer_id) REFERENCES users(id) ON DELETE CASCADE
);

-- Reviews
CREATE TABLE reviews (
    id TEXT PRIMARY KEY,
    order_id TEXT UNIQUE,
    gig_id TEXT NOT NULL,
    author_id TEXT NOT NULL,
    rating INTEGER NOT NULL CHECK (rating >= 1 AND rating <= 5),
    comment TEXT,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY(order_id) REFERENCES orders(id) ON DELETE CASCADE,
    FOREIGN KEY(gig_id) REFERENCES gigs(id) ON DELETE CASCADE,
    FOREIGN KEY(author_id) REFERENCES users(id) ON DELETE CASCADE
);

-- Messages
CREATE TABLE messages (
    id TEXT PRIMARY KEY,
    gig_id TEXT,
    sender_id TEXT NOT NULL,
    receiver_id TEXT NOT NULL,
    subject TEXT,
    body TEXT NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY(gig_id) REFERENCES gigs(id) ON DELETE SET NULL,
    FOREIGN KEY(sender_id) REFERENCES users(id) ON DELETE CASCADE,
    FOREIGN KEY(receiver_id) REFERENCES users(id) ON DELETE CASCADE
);
```
